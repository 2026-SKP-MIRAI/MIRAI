# engine/

## 목적
전원 공동 설계·기여하는 공통 기술 레이어. TDD 적용.
서비스들이 공유하는 파서·LLM·프롬프트 인프라를 제공.

## 구조
```
engine/
├── app/
│   ├── main.py       FastAPI 앱 진입점
│   ├── config.py     환경변수 (OPENROUTER_API_KEY 등)
│   ├── schemas.py    Pydantic 모델 (요청·응답 타입)
│   ├── parsers/      PDF → 텍스트 추출 (PyMuPDF)
│   ├── services/     LLM API 호출 (OpenRouter — google/gemini-2.5-flash)
│   ├── prompts/      프롬프트 템플릿 + 버전 관리
│   └── routers/      API 엔드포인트 (/resume, /interview 등)
├── tests/            pytest 테스트
│   └── fixtures/     테스트 데이터 (샘플 PDF, 예상 응답 JSON)
├── docs/             엔진 설계 문서
└── pyproject.toml    Python 프로젝트 설정
```

## 역할
- 엔진 불변식 4개를 실제로 구현하는 레이어
- 기여는 자율 — 특정 인원 지정 없음
- 서비스 skeleton이 엔진의 API 계약을 먼저 정의 (outside-in)
- 모든 구현은 테스트 먼저: Red → Green → Refactor

## 엔진 불변식
```
1. LLM API 호출은 반드시 engine/services/ 에서만
2. PDF 파싱은 반드시 engine/parsers/ 에서만
3. 각 service는 engine을 호출만 — 내부 직접 접근 금지
4. 테스트 없는 PR 머지 금지
```

## 파서 edge case 동작 계약

| 조건 | 예외 | HTTP | 비고 |
|------|------|------|------|
| 텍스트 0자 (빈 PDF) | `EmptyPDFError` (`ParseError` 하위) | **422** | 클라이언트 파일 문제, 재업로드 유도 |
| 텍스트 레이어 없는 이미지 PDF | `ImageOnlyPDFError` (`ParseError` 하위) | **422** | 클라이언트 파일 문제, 재업로드 유도 |
| 손상·암호화 PDF | `ParseError` | **400** | 클라이언트 파일 문제, 재업로드 유도 |
| 파일 크기 > 5MB | `FileSizeError` (`ParseError` 하위) | **400** | — |
| 페이지 수 > 10 | `PageLimitError` (`ParseError` 하위) | **400** | — |
| LLM API 오류 (OpenRouter) | `LLMError` | **500** | 서버 측 오류, 재시도 유도 |
| 예기치 않은 파싱 오류 | 미분류 예외 | **500** | 라이브러리 내부 오류 등 |

예외 계층:
```
ParseError (base)
├── EmptyPDFError       → 422
├── ImageOnlyPDFError   → 422
├── FileSizeError       → 400
└── PageLimitError      → 400
LLMError (독립)         → 500
```

> **TODO:** 예외 계층 세분화 — FileSizeError·PageLimitError는 파싱 전 검증 에러이므로 `ValidationError`로 분리 검토 (Beta 진입 시점에 결정)

- 파서(`app/parsers/`)는 예외를 던진다. 빈 문자열 반환 안 함.
- HTTP 변환은 `app/main.py` 전역 exception handler에서 처리.
- 서비스 레이어(`app/services/`)는 빈 문자열 입력 시 `LLMError` throw (파서가 걸러주므로 실제 도달하지 않아야 함).

## 규칙 변경 시 컨펌 필요
이 파일에 기재된 규칙과 충돌하는 변경을 하려는 경우, 작업을 진행하기 전에 반드시 사람(팀원)에게 먼저 확인을 받아야 한다.
