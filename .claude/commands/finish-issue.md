## 인수 형식 (모두 선택사항)

`$ARGUMENTS`는 다음 형식 중 하나거나 생략 가능하다:

| 형식 | 예시 | 동작 |
|------|------|------|
| 생략 | (없음) | 현재 브랜치에서 자동 감지 |
| 이슈번호만 | `15` | 이슈번호로 브랜치 탐색 |
| 6자리 패딩+짧은이름 | `000015-my-feature` | 직접 브랜치명 구성 |
| 짧은이름만 | `my-feature` | 브랜치 목록에서 매칭 |

## 실행 순서

### 1. 브랜치·이슈번호 확정

**인수 없는 경우 (워크트리 내 실행):**
```
git branch --show-current
```
결과 예: `feat/000015-pdf-upload` → type=`feat`, PADDED=`000015`, 짧은이름=`pdf-upload`, 이슈번호=`15`

**이슈번호(정수)만 있는 경우:**
```
git branch | grep {zero-padded 번호}
```
매칭된 브랜치명에서 type·짧은이름 추출.

**`000015-my-feature` 형식:**
```
git branch | grep 000015-my-feature
```

**짧은이름만 있는 경우:**
```
git branch | grep {짧은이름}
```

매칭된 브랜치가 없으면 오류 안내 후 중단.
- BRANCH = 확정된 브랜치명
- 이슈번호 = PADDED에서 정수 변환 (앞의 0 제거)
- WORKFOLDER = `docs/work/active/{PADDED}-{짧은이름}`

### 2. 변경사항 확인

다음 명령으로 변경사항을 파악한다:
```
git status
git diff --stat
```

### 3. 커밋 메시지 자동 생성

`git diff HEAD` 또는 `git diff`로 변경 내용을 분석한다.
변경된 파일과 내용을 바탕으로 커밋 메시지를 생성한다.

형식: `{type}: {변경 내용 요약} (#{이슈번호})`

예시:
- `feat: PDF 업로드 엔드포인트 구현 (#15)`
- `fix: 파서 예외 처리 추가 (#23)`
- `docs: API 인터페이스 명세 업데이트 (#31)`

### 4. 코드 간소화

커밋 전, 변경된 파일을 대상으로 불필요한 코드를 정리한다.
다음 항목을 확인하고 직접 수정한다:

- **디버그 출력 제거**: `print`, `console.log`, `logger.debug` 등 임시 출력
- **임시 코드 제거**: `# TODO: remove`, `# TEMP`, `# test용` 등 표시된 코드
- **잔여물 제거**: 하드코딩된 테스트 데이터, 임시 이미지·텍스트
- **중복 코드 리팩토링**: 3회 이상 반복되는 패턴은 함수·모듈로 추출

수정 사항이 있으면 변경 후 계속 진행한다.
수정 사항이 없으면:
```
✓ 코드 간소화 — 정리할 항목 없음
```

### 5. 코드 리뷰

`git diff HEAD` 또는 `git diff`로 변경사항 전체를 검토한다.
다음 항목을 중심으로 확인한다:

- **버그·로직 오류**: 잘못된 조건, 경계값, 예외 처리 누락
- **불변식 위반**: `CLAUDE.md` 기재 불변식 (LLM 호출 위치, PDF 파싱 위치 등)
- **테스트 누락**: 새 코드에 대응하는 테스트가 없는 경우
- **코드 품질**: 중복 코드, 불명확한 변수명, 불필요한 복잡성

발견된 문제를 아래 형식으로 출력한다:
```
🔍 코드 리뷰 결과

[ISSUE] {파일명}:{라인번호} — {문제 설명}
[ISSUE] ...

문제를 수정하고 싶으신가요? (수정 후 다시 /finish-issue를 실행하거나, 이대로 진행할 수 있습니다)
```

발견된 문제가 없으면:
```
✓ 코드 리뷰 통과 — 특이사항 없음
```

사용자가 수정하겠다고 하면 작업을 중단한다. 이대로 진행하겠다고 하면 다음 단계로 진행한다.

### 6. AC 체크

`{WORKFOLDER}/00_issue.md` 가 존재하면 파일을 읽어 `## 완료 기준` 섹션의 체크리스트를 추출한다.
존재하지 않으면 `gh issue view {이슈번호} --json body` 로 이슈 본문에서 완료 기준을 추출한다.

완료 기준 항목 각각에 대해, 변경된 코드·파일을 기준으로 충족 여부를 판단하고 다음 형식으로 출력한다:

```
📋 AC 체크

- [x] {완료 기준 1} — {근거: 어떤 변경으로 충족됐는지}
- [x] {완료 기준 2} — {근거}
- [ ] {완료 기준 3} — ⚠️ 미충족: {이유}
```

충족된 항목은 `{WORKFOLDER}/00_issue.md`의 `## 완료 기준` 섹션에서 해당 체크박스를 `- [x]`로 업데이트한다.
미충족 항목은 `- [ ]`로 유지한다.

미충족 항목이 있으면:
```
⚠️  미충족 AC가 있습니다. 계속 진행하시겠습니까? (y/n)
```

완료 기준이 없거나 추출 불가 시 이 단계를 건너뛴다.

### 7. .ai.md 업데이트 확인

작업 대상 디렉토리의 `.ai.md`가 최신 상태인지 확인한다.

```
⚠️  .ai.md 업데이트 확인
작업한 디렉토리의 .ai.md 에 변경사항이 반영됐나요?
- 새로 추가된 파일·모듈 기술
- 구조 변경 사항 반영
- 완료된 항목 업데이트

업데이트가 필요하면 먼저 수정 후 진행하세요.
```

사용자 확인 후 다음 단계로 진행한다.

### 8. 00_issue.md 작업 내역 작성

작업 폴더 존재 여부를 확인한다:
```
ls {WORKFOLDER}/ 2>/dev/null
```

**폴더가 존재하는 경우:**

다음 두 가지를 읽어 분석한다:
1. `{WORKFOLDER}/00_issue.md` — 이슈 배경·목적·완료 기준
2. `git diff` (또는 전체 브랜치 diff) — 실제 변경 내용

분석 결과를 `{WORKFOLDER}/00_issue.md` 의 `## 작업 내역` 섹션 아래에 추가한다.

작성 기준:
- 이슈에서 요구한 것과 실제 구현한 것을 연결해 서술
- 변경된 파일별로 무엇을 왜 변경했는지 설명
- 새로 추가된 기능·로직은 핵심 동작 위주로 설명
- 삭제·리팩토링은 이유 포함
- 기술적 결정 사항이 있으면 함께 기록

작성 완료 후 다음 단계(커밋)에서 함께 포함된다.

**폴더가 없는 경우:**
```
⚠️  작업 폴더 없음: {WORKFOLDER}
/start-issue 없이 작업을 시작한 경우입니다. PR 본문은 커밋 메시지 기반으로 작성합니다.
```

### 9. docs/work active → done 이동

PR 생성 전에 작업 폴더를 done으로 이동한다. 이 변경은 메인 커밋에 함께 포함된다.

**폴더가 존재하는 경우:**
```
mv {WORKFOLDER} docs/work/done/{PADDED}-{짧은이름}
```

**폴더가 없는 경우:** 이 단계를 건너뛴다.

### 10. 사용자 확인

`git status`와 `git diff --stat HEAD`를 실행해 변경 파일 목록을 파악한다.
커밋 메시지와 변경 파일 내역을 함께 보여주고 확인을 받는다:
```
커밋 메시지: "{생성된 메시지}"

변경 파일:
  M  {파일1}  (+{추가} / -{삭제})
  M  {파일2}  (+{추가} / -{삭제})
  A  {파일3}  (신규)
  ...

이대로 진행할까요? (수정이 필요하면 알려주세요)
```

확인 후 다음을 실행한다:
```
git add .
git commit -m "{커밋 메시지}"
```

변경사항이 이미 커밋된 경우 커밋 단계를 건너뛰고 push로 진행한다.

### 11. Push

다음 명령을 실행한다:
```
git push origin {BRANCH}
```

### 12. PR 생성

**완료문서가 있는 경우:**

`00_issue.md` 내용을 읽어 이슈 배경·목적을 2~3문장으로 요약하고,
`00_issue.md` 의 `## 작업 내역` 섹션, 그리고 단계 6에서 수행한 AC 체크 결과를 함께 PR 본문으로 구성한다:

```
gh pr create \
  --title "{커밋 메시지}" \
  --body "## 이슈 배경

{00_issue.md 내용 요약 — 2~3문장}

## 완료 기준 (AC)

- [x] {완료 기준 1}
- [x] {완료 기준 2}
- [ ] {완료 기준 3} _(미충족 시 표시)_

## 작업 내역

{00_issue.md의 ## 작업 내역 섹션 내용}

Closes #{이슈번호}"
```

AC 체크를 건너뛴 경우(완료 기준 없음) `## 완료 기준` 섹션은 생략한다.

**완료문서가 없는 경우:**
```
gh pr create \
  --title "{커밋 메시지}" \
  --body "Closes #{이슈번호}"
```

### 13. 완료 안내

다음 형식으로 출력한다:
```
✓ PR 생성 완료: {PR URL}

리뷰어로 멘토가 자동 지정됩니다.
PR 머지 후 /cleanup-issue {이슈번호} 을 실행하세요.
```
