## 인수 형식 (유연하게 지원)

`$ARGUMENTS`는 다음 형식 중 하나거나 생략 가능하다:

| 형식 | 예시 | 동작 |
|------|------|------|
| **생략** | (없음) | 전체 워크트리 스캔 → 머지된 이슈 자동 감지 후 일괄 정리 |
| 이슈번호만 | `15` | `git worktree list`에서 번호로 매칭 |
| 6자리 패딩+짧은이름 | `000015-my-feature` | 직접 워크트리 경로 구성 |
| 짧은이름만 | `my-feature` | `git worktree list`에서 이름으로 매칭 |

## 실행 순서

### 1. 워크트리·이슈번호 확정

**인수 없는 경우 — 자동 일괄 정리 모드:**

```
git worktree list
```

`.worktree/` 하위 항목을 모두 수집한다. 각 항목에서:
1. 이름 패턴 `{PADDED}-{짧은이름}`에서 이슈번호 추출
2. `gh issue view {번호} --json state` 로 상태 확인
3. CLOSED 상태인 것만 정리 대상으로 분류

정리 대상 목록을 출력하고 확인을 받는다:
```
머지 완료된 워크트리:
  - .worktree/000015-pdf-upload  (이슈 #15 CLOSED)
  - .worktree/000023-parser-fix  (이슈 #23 CLOSED)

위 2개를 모두 정리할까요? (y/n)
```

확인 후 각 항목에 대해 3~5단계를 순서대로 실행한다.

**이슈번호(정수)만 있는 경우:**
```
git worktree list | grep {zero-padded 번호}
```

**`000015-my-feature` 또는 `my-feature` 형식:**
```
git worktree list | grep {짧은이름}
```

매칭된 워크트리에서:
- WORKTREE = `.worktree/{PADDED}-{짧은이름}`
- 이슈번호 = PADDED에서 정수 변환

매칭 결과 없으면: "해당 워크트리를 찾을 수 없습니다. `git worktree list`로 확인하세요." 안내 후 중단.

### 2. 이슈 상태 확인 (단일 정리 모드)

```
gh issue view {이슈번호} --json state,title
```

이슈가 OPEN 상태이면 경고를 출력한다:
```
⚠️  이슈 #{이슈번호}가 아직 열려 있습니다.
PR이 머지됐는지 확인하세요. 머지 전 삭제 시 작업 내용이 날아갑니다.
계속 진행하시겠습니까?
```
사용자 확인 후 진행을 허용한다.

### 3. Worktree 삭제

다음 명령을 실행한다:
```
git worktree remove {WORKTREE}
```

실패하거나 존재하지 않으면:
```
이미 삭제됐거나 존재하지 않는 워크트리입니다. 브랜치 정리를 계속합니다.
```

### 4. 로컬 브랜치 삭제

다음 명령으로 브랜치를 찾는다:
```
git branch | grep {PADDED}-{짧은이름}
```

찾은 브랜치를 강제 삭제한다 (squash merge 시 `-d`로는 삭제 불가):
```
git branch -D {브랜치명}
```

없으면:
```
이미 삭제됐거나 존재하지 않는 브랜치입니다.
```

### 5. 리모트 브랜치 삭제

리모트 브랜치 존재 여부를 확인한다:
```
git ls-remote --heads origin {브랜치명}
```

존재하면 삭제한다:
```
git push origin --delete {브랜치명}
```

존재하지 않거나 실패하면:
```
리모트 브랜치가 이미 삭제됐거나 존재하지 않습니다.
```

### 6. 최종 상태 확인

다음 명령을 실행하고 결과를 출력한다:
```
git worktree list
```

정리 완료를 안내한다.
