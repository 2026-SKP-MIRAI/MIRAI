# MirAI MVP 개발 명세서

> **기준 문서:** `MirAI_proposal.md` §3 최소 기능 제품(MVP) 상세 명세
> **MVP 범위:** 자소서(PDF) 업로드 → 문서 분석 → 맞춤 예상 질문 리스트 생성

---

## 1. MVP 요약

| 항목 | 내용 |
|------|------|
| **핵심 가치** | "자소서를 올리면, 내 서류에서 나올 질문을 미리 알 수 있다" |
| **기능** | PDF 자소서 업로드 → 파싱·키워드/단락 추출 → 카테고리별 맞춤 예상 질문 생성 |
| **제외(다음 단계)** | 실시간 꼬리질문, 페르소나 면접관, 8축 역량 평가, 회원가입·결제 |

---

## 2. 필요한 기술 스택

### 2-1. 전체 구성

| 구분 | 기술 | 용도 | 비고 |
|------|------|------|------|
| **프론트엔드** | Next.js (App Router) | 단일 페이지, PDF 업로드 UI, 질문 리스트 표시 | 기존 프로젝트와 동일 시 유지 가능 |
| **스타일** | Tailwind CSS | 레이아웃·폼·카드 | v4 권장 |
| **언어** | TypeScript (strict) | 타입 안전성, API 계약 | 엄격 모드 |
| **PDF 처리** | 서버 측 PDF 파싱 라이브러리 | 업로드된 PDF → 텍스트 추출 | Node 환경 필수 |
| **AI** | Anthropic Claude API | 추출 텍스트 기반 맞춤 질문 생성 | 기존과 동일 시 비용·일관성 유리 |
| **스토리지 (MVP)** | 세션/메모리 또는 로컬 파일 | 업로드 파일·생성 질문 임시 보관 | DB·회원 미도입 권장 |

### 2-2. 기술 선택 이유

- **Next.js:** 풀스택 한 프로젝트에서 API 라우트로 PDF 수신·파싱·LLM 호출까지 처리 가능. 기존 mirai-personal과 동일 스택이면 통합·운영 일원화에 유리.
- **서버 측 PDF 파싱:** 브라우저만으로는 대용량·복잡 레이아웃 PDF 품질이 떨어질 수 있음. 서버에서 한 번만 파싱해 텍스트를 LLM에 넘기는 구조가 단순하고 품질 관리가 쉬움.
- **Claude API:** 한국어 프롬프트·긴 자소서 문맥 처리에 적합. 기존 면접관 루트와 동일 키·모델 사용 시 운영 단순.
- **MVP에 DB/회원 미도입:** 기획서 MVP는 "첫 경험·아하 모먼트"에 집중. 업로드 → 질문 생성 → 결과 표시만 있으면 되므로, 세션 또는 요청 단위 메모리로 충분. Free 3회/월 등 제한은 쿠키·로컬 카운트로 나중에 추가 가능.

### 2-3. PDF 파싱 라이브러리 후보

| 라이브러리 | 환경 | 특징 | MVP 적합성 |
|-----------|------|------|-----------|
| **pdf-parse** | Node.js | 단순 API, 텍스트만 추출, 의존성 적음 | ✅ 권장 |
| **pdfjs-dist (Mozilla PDF.js)** | Node/브라우저 | 풀 기능, 설정 부담 | 대안 |
| **unpdf** | Node (Rust 기반) | 빠르고 정확, 설치 복잡도 있을 수 있음 | 선택 |

> **권장:** `pdf-parse` — `npm i pdf-parse`, Buffer 넣으면 텍스트 반환. 자소서용으로 충분하고 구현이 빠름. (타입은 `@types/pdf-parse` 또는 직접 선언.)

## 3. 시스템 구성도

```
[사용자]
    |
    ▼
[업로드 화면] — PDF 선택 → [POST /api/resume/questions]
    |                              |
    |                              ├─ 1) PDF Buffer 수신
    |                              ├─ 2) pdf-parse로 텍스트 추출
    |                              ├─ 3) (선택) 추출 텍스트 길이/토큰 제한
    |                              ├─ 4) Claude API: 자소서 텍스트 → 맞춤 질문 생성
    |                              └─ 5) JSON 응답 { questions: [...] }
    |
    ▼
[결과 화면] ← 질문 리스트 표시 (카테고리별 구분 가능)
```

- **인증/DB 없음:** 요청 단위로 PDF 받고 → 파싱 → LLM → 응답 반환. 같은 세션에서 재요청 시 다시 업로드.

## 4. API 명세

### 4-1. POST /api/resume/questions

**역할:** PDF 자소서를 받아 맞춤 예상 질문 목록을 반환한다.

**요청**
- **Content-Type:** `multipart/form-data`
- **Body:** `file` (단일 파일, PDF만 허용 권장)
- **제한:** 파일 크기 상한 (예: 5MB), 페이지 수 상한 (예: 10페이지) — 선택이나 설정 권장

**응답 (성공, 200)**

```json
{
  "questions": [
    {
      "category": "직무 역량",
      "question": "OO 프로젝트에서 담당한 역할과 그 결과를 구체적으로 설명해 주세요."
    },
    {
      "category": "경험의 구체성",
      "question": "해당 경험에서 갈등이 있었다면 어떻게 해결했나요?"
    }
  ],
  "meta": {
    "extractedLength": 3200,
    "categoriesUsed": ["직무 역량", "경험의 구체성", "성과 근거", "기술 역량"]
  }
}
```

> **카테고리:** 기획서 기준 — 직무 역량, 경험의 구체성, 성과 근거, 기술 역량. (필드는 문자열 배열로 고정해 프론트에서 필터/정렬 가능하게.)

**응답 (에러)**
- `400`: 파일 없음, PDF 아님, 크기/페이지 초과
- `413`: 페이로드 과대 (설정한 경우)
- `500`: 파싱 실패, Claude API 오류 — 메시지는 한국어로 사용자 노출용 메시지 반환

## 5. 데이터·프롬프트

### 5-1. 파싱 출력

- PDF → **단일 텍스트 문자열**. 필요 시 `\n\n` 등으로 단락 구분 유지.
- **토큰/길이 제한:** Claude 입력 한도에 맞게 자소서 텍스트를 잘라서 사용 (예: 최대 8k~16k 문자 또는 모델 컨텍스트의 1/4 이하). 초과 분은 생략하고 "앞부분 기준으로 질문 생성" 등으로 프롬프트에 명시.

### 5-2. Claude 시스템 프롬프트 (초안)

- 역할: "자소서 기반 면접 예상 질문 생성 전문가."
- 입력: "아래는 지원자가 제출한 자기소개서 전문(또는 발췌)이다."
- 지시:
  - 문서에서 **프로젝트 경험, 직무 역량, 기술 키워드, 서술된 성과**를 반영해, 실제 면접관이 물어볼 가능성이 높은 질문만 생성한다.
  - 질문은 다음 **카테고리**에 맞춰 구분한다: 직무 역량, 경험의 구체성, 성과 근거, 기술 역량.
  - 각 카테고리당 2~5개, 총 8~20개 수준으로 생성 (숫자는 조정 가능).
  - 자소서에 없는 내용을 묻는 질문은 만들지 않는다.
- 출력 형식: JSON 배열 `[{ "category": "...", "question": "..." }]` 만 반환하도록 지시하고, 파싱 시 유효하지 않으면 재요청 또는 기본 메시지 처리.

## 6. 프론트엔드 (화면·플로우)

### 6-1. 화면 구성 (단일 페이지로 충분)

| 단계 | 화면 | 동작 |
|------|------|------|
| 1 | **업로드** | PDF 선택 (드래그 앤 드롭 또는 파일 선택), "질문 생성" 버튼 |
| 2 | **로딩** | "자소서를 분석하고 있습니다..." 등 메시지, 스피너 (비차단) |
| 3 | **결과** | 카테고리별로 질문 리스트 표시, "다른 자소서로 다시 하기" 시 업로드로 복귀 |

- **에러:** 400/500 시 토스트 또는 인라인 메시지로 한국어 안내 (예: "PDF만 업로드 가능합니다", "서버 오류가 났습니다. 잠시 후 다시 시도해 주세요.").

### 6-2. 상태

- `idle` → 업로드 대기
- `uploading` → 전송 중
- `processing` → 분석·질문 생성 중 (로딩 UI)
- `done` → 질문 리스트 표시
- `error` → 에러 메시지 표시

> (기존 면접관 페이지와 별도 라우트라면 `/resume` 등 단일 페이지로 위 상태만 관리.)

## 7. 개발 단계 제안

| 단계 | 작업 | 산출물 |
|------|------|--------|
| 1 | 프로젝트/라우트 준비 | Next.js 앱에 `/resume` 또는 MVP 전용 진입점, Tailwind 적용 |
| 2 | PDF 업로드 API | `multipart/form-data` 수신, 파일 타입/크기 검증, `pdf-parse`로 텍스트 추출, 추출 텍스트만 로그로 확인 |
| 3 | Claude 연동 | 추출 텍스트를 시스템/유저 메시지에 넣고, 위 §5-2 프롬프트로 질문 생성, JSON 파싱 및 검증 |
| 4 | API 라우트 완성 | `POST /api/resume/questions` 응답 형식 (§4-1) 고정, 에러 코드·한국어 메시지 정리 |
| 5 | 업로드 UI | PDF 선택/드롭, "질문 생성" 버튼, 로딩 상태 |
| 6 | 결과 UI | `questions` 배열을 카테고리별로 표시, "다시 하기" 버튼 |
| 7 | (선택) 파일 크기·페이지 제한, 에러 처리 다듬기 | 400/413 응답, 사용자 메시지 정리 |

## 8. MVP에서 하지 않는 것

- 회원가입·로그인
- 결제·구독 (Free 3회/Pro 등)
- 자소서·생성 질문 DB 저장
- 실시간 꼬리질문, 페르소나 면접관, 8축 역량 평가
- 오디오·영상 녹화

---

## 9. 의존성 추가 요약

- **필수:** `pdf-parse` (PDF → 텍스트)
- **이미 있음:** Next.js, React, Tailwind, Anthropic SDK, TypeScript
- **타입:** `pdf-parse`에 `@types`가 없으면 `declare module 'pdf-parse'` 등으로 타입 선언

## 10. 환경 변수

- `ANTHROPIC_API_KEY` — Claude 호출 (기존과 동일)
- `CLAUDE_MODEL` — (선택) 사용할 모델. 미설정 시 기존과 동일한 기본값 사용.
