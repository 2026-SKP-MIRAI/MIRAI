# MirAI MVP 개발 명세서

> **기준 문서:** `MVP_proposal.md` §3 최소 기능 제품(MVP) 상세 명세
> **MVP 범위:** 자소서(PDF) 업로드 → 문서 분석 → 맞춤 예상 질문 리스트 생성

---

## 1. MVP 요약

| 항목 | 내용 |
|------|------|
| **핵심 가치** | "자소서를 올리면, 내 서류에서 나올 질문을 미리 알 수 있다" |
| **기능** | PDF 자소서 업로드 → 파싱·키워드/단락 추출 → 카테고리별 맞춤 예상 질문 생성 |
| **제외(다음 단계)** | 실시간 꼬리질문, 페르소나 면접관, 8축 역량 평가, 회원가입·결제 |

---

## 2. 기술 스택

### 엔진 (engine/ — FastAPI, Python)

| 구분 | 기술 | 용도 |
|------|------|------|
| **서버** | Python FastAPI | API 엔드포인트 |
| **언어** | Python 3.12+ | — |
| **타입 시스템** | Pydantic v2 (`schemas.py`) | 요청·응답 모델 |
| **설정 관리** | pydantic-settings (`config.py`) | 환경변수 |
| **AI** | `anthropic` Python SDK | LLM 호출 (`app/services/`에서만) |
| **PDF 처리** | PyMuPDF (`fitz`) | 텍스트 추출 (`app/parsers/`에서만) |
| **테스트** | pytest | 단위·통합 테스트 |

### 서비스 (services/ — Next.js, TypeScript)

| 구분 | 기술 | 용도 |
|------|------|------|
| **프론트엔드** | Next.js (App Router) | PDF 업로드 UI, 질문 리스트 표시 |
| **스타일** | Tailwind CSS v4 | 레이아웃·폼·카드 |
| **언어** | TypeScript (strict) | 타입 안전성 |
| **스토리지 (MVP)** | 세션/메모리 | DB·회원 미도입 |
| **테스트** | Vitest | 단위 테스트 |

> Better Auth·Prisma·PostgreSQL은 MVP 범위 외 (회원가입·DB 저장 미도입)

### 통신

```
[유저] → [Next.js 서비스] → HTTP REST → [FastAPI 엔진 (내부 전용)]
```

- `ENGINE_BASE_URL` 환경변수, 타임아웃 30초
- 에러: FastAPI → JSON → 서비스에서 유저 메시지로 변환
- **MVP:** 인증 없음. 요청 단위 stateless 처리.

> **Beta 확장 시:** Better Auth 인증 레이어는 Phase 1 (Beta) 진입 시 서비스(Next.js)에 추가한다. 엔진은 변경 없이 내부 전용 유지.

## 3. 개발 원칙

### 엔진 — TDD (Test-Driven Development)

파서·LLM·프롬프트 레이어를 명확하게 분리. **구현 전 테스트를 먼저 작성**한다. AC = 첫 번째 테스트. Red → Green → Refactor.

### 서비스 — DDD 1순위, TDD 2순위

도메인(면접 코칭)이 기술 구현보다 앞선다. 구현 전 테스트를 먼저 작성한다.

### 아키텍처 불변식 (위반 시 CI 차단)

```
1. 인증은 서비스(Next.js)에서만 — 엔진은 인증 로직 없이 내부 호출만 수신
2. 외부 AI API 호출은 엔진에서만 — 서비스가 직접 LLM을 호출하지 않는다
3. 서비스 간 직접 통신 금지 — 각 서비스는 독립적, 공유 로직은 엔진으로
4. DB는 서비스가 소유 — 엔진은 stateless, 데이터 저장은 서비스 책임
5. 테스트 없는 PR은 머지 금지
```

> MVP는 인증·DB가 없으므로 불변식 1·4는 "MVP 범위 내 해당 없음"이나, Beta 진입 시 자동 적용된다.

## 4. 시스템 구성도

```
[사용자]
    |
    ▼
[업로드 화면 — Next.js 서비스]
    | PDF 선택
    ▼
POST /api/resume/questions (Next.js API 라우트)
    | HTTP REST (ENGINE_BASE_URL)
    ▼
[FastAPI 엔진]
    ├─ 1) PDF 수신
    ├─ 2) PyMuPDF(fitz)로 텍스트 추출 (app/parsers/)
    ├─ 3) 추출 텍스트 길이/토큰 제한
    ├─ 4) Claude API: 자소서 텍스트 → 맞춤 질문 생성 (app/services/)
    └─ 5) JSON 응답 { questions: [...] }
    |
    ▼
[결과 화면 — Next.js 서비스] ← 질문 리스트 표시 (카테고리별 구분)
```

- **인증/DB 없음(MVP):** 요청 단위로 PDF 받고 → 파싱 → LLM → 응답 반환. 같은 세션에서 재요청 시 다시 업로드.

## 5. API 명세

### 5-1. POST /api/resume/questions

**역할:** PDF 자소서를 받아 맞춤 예상 질문 목록을 반환한다.

**요청**
- **Content-Type:** `multipart/form-data`
- **Body:** `file` (단일 파일, PDF만 허용 권장)
- **제한:** 파일 크기 상한 (예: 5MB), 페이지 수 상한 (예: 10페이지) — 선택이나 설정 권장

**응답 (성공, 200)**

```json
{
  "questions": [
    {
      "category": "직무 역량",
      "question": "OO 프로젝트에서 담당한 역할과 그 결과를 구체적으로 설명해 주세요."
    },
    {
      "category": "경험의 구체성",
      "question": "해당 경험에서 갈등이 있었다면 어떻게 해결했나요?"
    }
  ],
  "meta": {
    "extractedLength": 3200,
    "categoriesUsed": ["직무 역량", "경험의 구체성", "성과 근거", "기술 역량"]
  }
}
```

> **카테고리:** 기획서 기준 — 직무 역량, 경험의 구체성, 성과 근거, 기술 역량. (필드는 문자열 배열로 고정해 Beta에서 필터/정렬 확장 가능.)

**응답 (에러)**
- `400`: 파일 없음, PDF 아님, 크기/페이지 초과, 손상·암호화 PDF — 클라이언트 파일 문제, 재업로드 유도
- `413`: 페이로드 과대 (설정한 경우)
- `422`: 빈 PDF (텍스트 0자), 이미지 전용 PDF — 형식은 유효하나 처리 불가, 재업로드 유도
- `500`: Claude API 오류, 예기치 않은 파싱 오류 — 서버 측 오류, 메시지는 한국어로 반환

## 6. 데이터·프롬프트

### 6-1. 파싱 출력

- PDF → **단일 텍스트 문자열**. 필요 시 `\n\n` 등으로 단락 구분 유지.
- **토큰/길이 제한:** Claude 입력 한도에 맞게 자소서 텍스트를 잘라서 사용 (예: 최대 8k~16k 문자 또는 모델 컨텍스트의 1/4 이하). 초과 분은 생략하고 "앞부분 기준으로 질문 생성" 등으로 프롬프트에 명시.

### 6-2. Claude 시스템 프롬프트 (초안)

> 프롬프트 파일 위치: `engine/app/prompts/` — 버전 관리·교체 가능하도록 분리.

- 역할: "자소서 기반 면접 예상 질문 생성 전문가."
- 입력: "아래는 지원자가 제출한 자기소개서 전문(또는 발췌)이다."
- 지시:
  - 문서에서 **프로젝트 경험, 직무 역량, 기술 키워드, 서술된 성과**를 반영해, 실제 면접관이 물어볼 가능성이 높은 질문만 생성한다.
  - 질문은 다음 **카테고리**에 맞춰 구분한다: 직무 역량, 경험의 구체성, 성과 근거, 기술 역량.
  - 각 카테고리당 2~5개, 총 8~20개 수준으로 생성 (숫자는 조정 가능).
  - 자소서에 없는 내용을 묻는 질문은 만들지 않는다.
- 출력 형식: JSON 배열 `[{ "category": "...", "question": "..." }]` 만 반환하도록 지시하고, 파싱 시 유효하지 않으면 재요청 또는 기본 메시지 처리.

## 7. 프론트엔드 (화면·플로우)

### 7-1. 화면 구성 (단일 페이지로 충분)

| 단계 | 화면 | 동작 |
|------|------|------|
| 1 | **업로드** | PDF 선택 (드래그 앤 드롭 또는 파일 선택), "질문 생성" 버튼 |
| 2 | **로딩** | "자소서를 분석하고 있습니다..." 등 메시지, 스피너 (비차단) |
| 3 | **결과** | 카테고리별로 질문 리스트 표시, "다른 자소서로 다시 하기" 시 업로드로 복귀 |

- **에러:** 400/500 시 토스트 또는 인라인 메시지로 한국어 안내 (예: "PDF만 업로드 가능합니다", "서버 오류가 났습니다. 잠시 후 다시 시도해 주세요.").

### 7-2. 상태

- `idle` → 업로드 대기
- `uploading` → 전송 중
- `processing` → 분석·질문 생성 중 (로딩 UI)
- `done` → 질문 리스트 표시
- `error` → 에러 메시지 표시

> (기존 면접관 페이지와 별도 라우트라면 `/resume` 등 단일 페이지로 위 상태만 관리.)

## 8. 개발 단계 제안

**순서: 테스트 먼저 작성 → Red → Green → Refactor**

| 단계 | 작업 | 산출물 |
|------|------|--------|
| 1 | 엔진 파서 구현 | `engine/app/parsers/` — pytest 작성 먼저 → PyMuPDF로 PDF → 텍스트 추출 |
| 2 | 엔진 LLM 서비스 구현 | `engine/app/services/` — pytest 작성 먼저 → anthropic SDK로 질문 생성 |
| 3 | 엔진 API 라우트 완성 | `POST /api/resume/questions` 응답 형식 (§5-1), 에러 코드·한국어 메시지 |
| 4 | 서비스 API 라우트 | Next.js API 라우트 — PDF 수신 → 엔진 HTTP 호출 → 응답 반환 |
| 5 | 업로드 UI | PDF 선택/드롭, "질문 생성" 버튼, 로딩 상태 (Vitest 테스트) |
| 6 | 결과 UI | `questions` 배열을 카테고리별로 표시, "다시 하기" 버튼 |
| 7 | (선택) 파일 크기·페이지 제한, 에러 처리 다듬기 | 400/413 응답, 사용자 메시지 정리 |

## 9. MVP에서 하지 않는 것 (Beta 확장 경로)

**MVP 범위 외 (다음 단계):**
- 회원가입·로그인 (Better Auth)
- 결제·구독 (Free 3회/Pro 등)
- 자소서·생성 질문 DB 저장 (Prisma + PostgreSQL)
- 실시간 꼬리질문, 페르소나 면접관, 8축 역량 평가
- 오디오·영상 녹화

**Beta 확장 경로 (Phase 1~4):**

| Phase | 추가 기능 | 추가 기술 |
|-------|---------|---------|
| Phase 1 | 패널 면접 채팅 + 꼬리질문 엔진 | 기존 스택 그대로 (엔진 routers/ 확장) |
| Phase 2 | 8축 역량 평가 리포트 | 기존 스택 그대로 |
| Phase 3 | 연습 모드 피드백·재답변 + 서류 진단 | Better Auth 인증 + Prisma + PostgreSQL |
| Phase 4 | AI 아바타 면접 (TTS/STT) | S3, CloudFront, 오디오 처리 |

> MVP 코드는 Phase 1 확장을 막지 않도록 설계한다. 엔진 routers/ 구조와 schemas.py 응답 타입을 확장 가능하게 유지.
