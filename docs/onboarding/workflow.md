# MirAI 워크플로우 가이드

> 처음 기여하는 팀원을 위한 가이드. 5분이면 충분하다.

## 처음 한 번만: 환경 설정

```bash
gh auth login
```

GitHub 계정 인증. 이거 한 번만 하면 이슈 생성, PR, 코드 푸시 전부 된다.
Projects 보드 자동화는 별도 설정 없이 알아서 동작한다.

---

## 한 줄 요약

```
Ready 이슈 픽업 → 브랜치 생성 → 구현 (테스트 먼저) → PR → 멘토 리뷰 → 머지
```

---

## 보드 컬럼

| 컬럼 | 의미 | 누가 이동시키나 |
|------|------|-----------------|
| **Backlog** | 이슈 생성 직후. 아직 시작하면 안 됨 | 자동 (이슈 생성 시) |
| **Ready** | 멘토 확인 완료. 지금 바로 착수 가능 | 멘토 |
| **In Progress** | 내가 작업 중 | 자동 (이슈 Assign 시) |
| **In Review** | PR 올린 상태. 멘토 리뷰 대기 | 직접 이동 |
| **Done** | 머지 완료 | 자동 (PR 머지 시) |
| **Drop** | 안 하기로 결정 | 멘토 또는 본인 |

> **내가 할 수 있는 이슈 = Ready 컬럼에 있는 것.**
> Backlog는 멘토가 Ready로 옮겨줄 때까지 기다린다.

---

## 1단계: 내 이슈 확인

```bash
gh issue list --assignee @me
```

또는 GitHub Projects 보드에서 **Ready** 컬럼 확인.

이슈가 없으면 멘토에게 요청하거나, 본인이 이슈를 만들어 멘토에게 Assign 요청.

---

## 2단계: 브랜치 생성

```bash
git checkout -b feat/15-pdf-upload    # 기능 구현
git checkout -b fix/23-parser-error   # 버그 수정
git checkout -b docs/31-interface-doc # 문서 작업
```

규칙: `타입/이슈번호-짧은설명`

---

## 3단계: 구현 (테스트 먼저)

```
이슈의 AC(인수 조건)를 먼저 읽는다
  ↓
AC를 테스트 코드로 작성 (Red)
  ↓
테스트 통과하는 코드 작성 (Green)
  ↓
리팩터링 (Refactor)
  ↓
해당 디렉토리 .ai.md 업데이트
```

**AC = 첫 번째 테스트.** 이슈에 AC가 없으면 멘토에게 먼저 물어본다.

---

## 4단계: PR 생성

```bash
git add .
git commit -m "feat: PDF 업로드 엔드포인트 구현 (#15)"
git push origin feat/15-pdf-upload
gh pr create --title "feat: PDF 업로드 엔드포인트 구현 (#15)"
```

PR 본문에 반드시 포함:
```
Closes #15
```
이 한 줄이 있어야 머지 시 이슈가 자동으로 닫힌다.

---

## 5단계: 리뷰 기다리기

- Reviewer로 멘토가 자동 지정됨
- 멘토가 코멘트 남기면 수정 후 다시 push
- 추가 커밋은 같은 브랜치에 push하면 PR에 자동 반영

---

## 6단계: 머지

멘토 승인 후 멘토가 Squash merge.
머지되면 브랜치 자동 삭제, 이슈 자동 Close, Projects 보드 Done 이동.

---

## 커밋 메시지 규칙

```
feat: 새 기능
fix: 버그 수정
docs: 문서만 변경
refactor: 동작 변경 없는 코드 개선
test: 테스트만 변경
```

예시: `feat: 자소서 PDF 파싱 구현 (#12)`

---

## 이슈 작성법 (본인이 만들 때)

**완료 기준이 없는 이슈는 시작할 수 없다.**

이슈 종류에 따라 템플릿을 선택한다:

### feature — 사용자에게 가치를 주는 기능

```markdown
## 사용자 관점 목표
내 자소서에서 나올 질문을 미리 알고, 면접을 자신 있게 준비할 수 있다

## 배경
현재 면접 준비는 일반적인 예상 질문에 의존한다. 자소서 기반 맞춤 질문이 없다.

## 완료 기준
- [ ] 자소서를 올리면 내 경험에 맞는 질문이 화면에 나온다
- [ ] 잘못된 파일을 올려도 당황하지 않는다 (안내 메시지)

## 개발 체크리스트
- [ ] 테스트 코드 포함
- [ ] 해당 디렉토리 .ai.md 최신화
- [ ] 불변식 위반 없음
```

> 완료 기준은 API 스펙이 아니라 **사용자가 경험하는 가치**로 쓴다.
> 클로드 코드에게 이 이슈를 시키면 완료 기준이 첫 번째 테스트가 된다.

### chore — 팀/에이전트의 환경을 개선하는 작업

```markdown
## 목적
불변식 위반이 자동으로 차단된다 (pre-commit 훅)

## 배경
지금은 글로 쓴 규칙에만 의존한다. 실수로 위반해도 PR 전까지 모른다.

## 완료 기준
- [ ] engine/services/ 밖에서 LLM 호출 시 커밋이 차단된다
```

---

## 이슈 Drop (안 하기로 결정)

이슈를 구현하지 않기로 판단되면:
1. 이슈에 이유를 코멘트로 남긴다
2. Projects 보드에서 **Drop** 컬럼으로 이동
3. 이슈는 Close (not planned)

> Drop ≠ 실패. "이 시점에 이 이슈는 우선순위 밖"이라는 명시적 결정이다.

---

## 막혔을 때

1. 이슈에 코멘트로 질문 남기기 (멘토가 비동기로 확인)
2. PR을 Draft로 열고 `[WIP]` 표시
3. 진행 불가 시 이슈에 `blocked` 라벨 추가
