# MirAI 워크플로우 가이드

> 처음 기여하는 팀원을 위한 가이드. 5분이면 충분하다.

## 처음 한 번만: 환경 설정

```bash
gh auth login
```

GitHub 계정 인증. 이거 한 번만 하면 이슈 생성, PR, 코드 푸시 전부 된다.
Projects 보드 자동화는 별도 설정 없이 알아서 동작한다.

---

## 한 줄 요약

```
Ready 이슈 픽업 → 워크트리+브랜치 생성 → 구현 (테스트 먼저) → PR → 멘토 리뷰 → 머지 → 워크트리 삭제
```

---

## 보드 컬럼

| 컬럼 | 의미 | 누가 이동시키나 |
|------|------|-----------------|
| **Backlog** | 이슈 생성 직후. 아직 시작하면 안 됨 | 자동 (이슈 생성 시) |
| **Ready** | 멘토 확인 완료. 지금 바로 착수 가능 | 멘토 |
| **In Progress** | 내가 작업 중 | 자동 (이슈 Assign 시) |
| **In Review** | PR 올린 상태. 멘토 리뷰 대기 | 직접 이동 |
| **Done** | 머지 완료 | 자동 (PR 머지 시) |
| **Drop** | 안 하기로 결정 | 멘토 또는 본인 |

> **내가 할 수 있는 이슈 = Ready 컬럼에 있는 것.**
> Backlog는 멘토가 Ready로 옮겨줄 때까지 기다린다.

---

## 1단계: 내 이슈 확인

```bash
gh issue list --assignee @me
```

또는 GitHub Projects 보드에서 **Ready** 컬럼 확인.

이슈가 없으면 멘토에게 요청하거나, 본인이 이슈를 만들어 멘토에게 Assign 요청.

---

## 2단계: 워크트리 + 브랜치 생성

워크트리를 쓰면 브랜치마다 독립된 디렉토리에서 작업할 수 있다.
main을 건드리지 않고 여러 이슈를 동시에 진행할 때 유용하다.

```bash
# 형식: git worktree add <경로> -b <브랜치명>
git worktree add .worktree/000015-pdf-upload -b feat/000015-pdf-upload    # 기능 구현
git worktree add .worktree/000023-parser-error -b fix/000023-parser-error # 버그 수정
git worktree add .worktree/000031-interface-doc -b docs/000031-interface-doc # 문서 작업
```

워크트리 생성 후 해당 디렉토리로 이동해서 작업한다:

```bash
cd .worktree/000015-pdf-upload
```

브랜치 규칙: `타입/이슈번호(6자리 0패딩)-짧은설명`
경로 규칙: `.worktree/이슈번호(6자리 0패딩)-짧은설명` (레포 루트 내부, `.gitignore`로 추적 제외)

> 워크트리·브랜치·작업폴더 세 곳의 이름을 동일하게 맞춘다.
> 이슈 #15 → `.worktree/000015-pdf-upload` / `feat/000015-pdf-upload` / `docs/work/active/000015-pdf-upload`

> 워크트리가 없던 시절엔 `git checkout -b`로 브랜치만 생성했다. 이제는 워크트리 방식을 사용한다.

---

## 3단계: 작업 폴더 생성

이슈를 시작하면 `docs/work/active/`에 폴더를 만든다.

```bash
mkdir docs/work/active/000015-pdf-upload
```

규칙: `이슈번호(6자리 0패딩)-짧은설명`
예시: 이슈 #15 → `000015-pdf-upload`, 이슈 #123 → `000123-parser-error`

이 폴더에 작업 중 생성되는 문서(분석 결과, 설계 메모, 산출물 등)를 담는다.
PR이 머지되면 `docs/work/done/`으로 자동 이동된다.

> 폴더를 만들지 않아도 작업은 가능하다. 산출 문서가 있을 때만 만든다.

---

## 4단계: 구현 (테스트 먼저)

```
이슈의 AC(인수 조건)를 먼저 읽는다
  ↓
AC를 테스트 코드로 작성 (Red)
  ↓
테스트 통과하는 코드 작성 (Green)
  ↓
리팩터링 (Refactor)
  ↓
해당 디렉토리 .ai.md 업데이트
```

**AC = 첫 번째 테스트.** 이슈에 AC가 없으면 멘토에게 먼저 물어본다.

---

## 5단계: PR 생성

```bash
git add .
git commit -m "feat: PDF 업로드 엔드포인트 구현 (#15)"
git push origin feat/000015-pdf-upload
gh pr create --title "feat: PDF 업로드 엔드포인트 구현 (#15)"
```

PR 본문에 반드시 포함:
```
Closes #15
```
이 한 줄이 있어야 머지 시 이슈가 자동으로 닫힌다.

---

## 6단계: 리뷰 기다리기

- Reviewer로 멘토가 자동 지정됨
- 멘토가 코멘트 남기면 수정 후 다시 push
- 추가 커밋은 같은 브랜치에 push하면 PR에 자동 반영

---

## 7단계: 머지

멘토 승인 후 멘토가 Squash merge.
머지되면 브랜치 자동 삭제, 이슈 자동 Close, Projects 보드 Done 이동.

---

## 8단계: 워크트리 삭제

머지 확인 후 로컬 워크트리를 정리한다.

```bash
# 레포 루트로 복귀 후 실행
git worktree remove .worktree/000015-pdf-upload

# 로컬 브랜치 정리 (이미 머지됐으므로 -d로 안전 삭제)
git branch -d feat/000015-pdf-upload
```

워크트리 목록 확인:
```bash
git worktree list
```

> 머지 전에 워크트리를 지우면 작업 내용이 날아간다. **반드시 머지 확인 후** 삭제.

---

## 커밋 컨벤션

### 원칙: 이슈 단위 1커밋

- **브랜치당 커밋 1개**를 기본으로 한다. docs/work 이동, `.ai.md` 업데이트 등 부수 작업도 메인 커밋에 포함한다.
- 멘토는 **Squash merge**로 머지하므로, 최종적으로 main에는 이슈당 커밋 1개만 남는다.
- 작업 중 중간 커밋이 필요하면 해도 되지만, `/finish-issue`는 하나로 묶는다.

### 커밋 메시지 형식

```
{type}: {변경 내용 요약} (#{이슈번호})
```

**type 종류:**
```
feat:     새 기능
fix:      버그 수정
docs:     문서만 변경
refactor: 동작 변경 없는 코드 개선
test:     테스트만 변경
chore:    빌드·설정·도구 등 비기능 변경
```

**예시:**
- `feat: 자소서 PDF 파싱 구현 (#12)`
- `fix: 파서 예외 처리 추가 (#23)`
- `refactor: 엄브렐라 레포 전환 — engine, services를 독립 레포 submodule로 분리 (#17)`

### 하지 않는 것

- `chore: docs/work done 이동` 같은 부수 작업 전용 커밋을 만들지 않는다.
- 이슈와 무관한 잡다한 정리를 끼워 넣지 않는다. 별도 이슈로 분리한다.

## 이슈 작성법 (본인이 만들 때)

**완료 기준이 없는 이슈는 시작할 수 없다.**

이슈 종류에 따라 템플릿을 선택한다:

### feature — 사용자에게 가치를 주는 기능

```markdown
## 사용자 관점 목표
내 자소서에서 나올 질문을 미리 알고, 면접을 자신 있게 준비할 수 있다

## 배경
현재 면접 준비는 일반적인 예상 질문에 의존한다. 자소서 기반 맞춤 질문이 없다.

## 완료 기준
- [ ] 자소서를 올리면 내 경험에 맞는 질문이 화면에 나온다
- [ ] 잘못된 파일을 올려도 당황하지 않는다 (안내 메시지)

## 개발 체크리스트
- [ ] 테스트 코드 포함
- [ ] 해당 디렉토리 .ai.md 최신화
- [ ] 불변식 위반 없음
```

> 완료 기준은 API 스펙이 아니라 **사용자가 경험하는 가치**로 쓴다.
> 클로드 코드에게 이 이슈를 시키면 완료 기준이 첫 번째 테스트가 된다.

### chore — 팀/에이전트의 환경을 개선하는 작업

```markdown
## 목적
불변식 위반이 자동으로 차단된다 (pre-commit 훅)

## 배경
지금은 글로 쓴 규칙에만 의존한다. 실수로 위반해도 PR 전까지 모른다.

## 완료 기준
- [ ] engine/services/ 밖에서 LLM 호출 시 커밋이 차단된다
```

---

## 이슈 Drop (안 하기로 결정)

이슈를 구현하지 않기로 판단되면:
1. 이슈에 이유를 코멘트로 남긴다
2. Projects 보드에서 **Drop** 컬럼으로 이동
3. 이슈는 Close (not planned)

> Drop ≠ 실패. "이 시점에 이 이슈는 우선순위 밖"이라는 명시적 결정이다.

---

## 막혔을 때

1. 이슈에 코멘트로 질문 남기기 (멘토가 비동기로 확인)
2. PR을 Draft로 열고 `[WIP]` 표시
3. 진행 불가 시 이슈에 `blocked` 라벨 추가
