## 0. 전제

**두 가지를 동시에 목표로 한다: "풀스택 빌더" + "하네스 엔지니어"**

| 역량 | 풀스택 빌더 | 하네스 엔지니어 |
|------|------------|--------------|
| 핵심 | 서비스를 처음부터 끝까지 직접 만드는 역량 | 에이전트가 올바르게 작동할 수 있는 환경 설계 역량 |
| 산출물 | 동작하는 서비스 (프론트·백·배포) | AC + 아키텍처 불변식 + 피드백 루프 |
| 포트폴리오 | "혼자 서비스 하나 완성" | "에이전트 팀을 운영해본 경험" |

**왜 하네스 엔지니어인가:**
OpenAI는 5개월, 엔지니어 3~7명, 수동 코드 0줄로 ~100만 줄짜리 제품을 만들었다. 비결은 코딩 실력이 아니었다 — 에이전트가 스스로 검증할 수 있는 환경 설계였다. 한국 신입 공채가 사실상 없는 지금, 이 역량이 "코딩 잘하는 신입"보다 차별점이 된다.

**4주의 실질적 목표:**
1. 팀 엔진 공동 설계 경험
2. 혼자 서비스 하나 처음부터 끝까지 만든 경험
3. 포트폴리오 소재 1개

---

## 1. 하네스 엔지니어란

> **"Humans steer. Agents execute."** — OpenAI, Harness Engineering, Feb 2026

하네스 엔지니어는 코드를 직접 쓰는 사람이 아니라, **에이전트가 올바른 코드를 쓸 수 있는 환경을 설계하는 사람**이다.

### 4가지 핵심 실천

| 실천 | 내용 | MirAI 적용 |
|------|------|----------|
| 레포 = 지식의 SOT | 레포에 없으면 에이전트에겐 없는 것 | 카톡/노션 결정사항 → 레포로 |
| 에이전트 자기검증 | 에이전트가 스스로 결과를 확인할 수 있는 도구 | pytest + CI + 로컬 API 확인 |
| 아키텍처 불변식 기계 강제 | 경계 위반을 사람이 아닌 도구가 잡음 | pre-commit + 린터 |
| 실패 → 환경 개선 루프 | 에이전트 실패 = 환경 설계 실패 신호 | 실패 로그 → AGENTS.md 업데이트 |

### MirAI = 팀의 하네스 실습장

별도 스터디가 없다. MirAI 자체가 교실이다. 엔진을 함께 만들고, 각자 서비스를 혼자 만드는 과정 전체가 하네스 설계 실습이다.

## 2. 채용시장 현실

> 잡코리아·사람인+점핏·LinkedIn·Glassdoor 직접 확인 (2026-03-01) / 전체: `dev_job_market_2026.md`

| 직군 | 전체 공고 | 신입 공고 | 현실 |
|------|---------|---------|------|
| 백엔드 | 2,682건 | 145건 | 가장 현실적인 진입 경로 |
| 프론트엔드 | 1,780건 | 170건 | 포트폴리오 없으면 불가 |
| DS/ML 신입 | 376건 | **11건** | 사실상 없음 |
| 전체 평균 | 10만 건 | **0.8%** | 기업은 4~7년차 49.7% 선호 |

**차별점:** "AI 활용해봤습니다"는 이미 기본값이다. "에이전트 팀이 쓰는 환경을 직접 설계했습니다"는 아직 드물다.

## 4. 멘토(이왕원) 역할

### 해야 할 것

| 항목 | 주기 | 구체적 행동 |
|------|------|-----------|
| AC 품질 리뷰 | 매 Issue | "이 AC가 에이전트 실행 가능한가?" |
| 실패 진단 | 발생 시 | "어떤 정보가 레포에 없었나?" 함께 분석 |
| 주간 회고 | 매주 금 | 팀 학습 → AGENTS.md 인코딩 |
| 하네스 시범 | Week 2 | 전체 사이클 직접 시연 (환경 설계 → 에이전트 실행 → 검증) |

### 하지 말아야 할 것

| 금지 | 이유 |
|------|------|
| 멘티 대신 코드 짜주기 | 환경 설계를 배워야 함 |
| AC 없는 PR 묵인 | AC 없으면 에이전트도, 리뷰도 없다 |
| 에이전트 실패 시 직접 해결 | "뭐가 레포에 없었나"를 멘티가 찾아야 함 |
| 서비스 범위 확장 묵인 | 자소서→질문 1번 기능이 안 되면 나머지는 의미 없다 |
