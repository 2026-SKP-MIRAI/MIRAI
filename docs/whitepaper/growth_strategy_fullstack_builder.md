## 0. 전제

풀스택 빌더와 하네스 엔지니어는 **다른 역할**이다. 엔진은 멘토·멘티 전원이 함께 만든다. 단, 다음 영역은 멘토가 주도한다:

| 멘토 주도 영역 | 내용 |
|--------------|------|
| **프로젝트 세팅** | 레포 구조, AGENTS.md 초기 설계, 온보딩 환경 |
| **에이전트 동작 환경** | CI, pre-commit, 불변식 정의, exec-plans 구조 |
| **에이전트 정의** | 어떤 에이전트가 어떤 역할을 하는지 설계 |
| **PM** | 프로덕트 방향, 우선순위, 스코프 통제 |

**역할의 의미:**

| 역량 | 풀스택 빌더 | 하네스 엔지니어 |
|------|------------|--------------|
| 핵심 | 서비스를 처음부터 끝까지 직접 만드는 역량 | 에이전트가 올바르게 작동할 수 있는 환경 설계 역량 |
| 산출물 | 동작하는 서비스 (프론트·백·배포) | AC + 아키텍처 불변식 + 피드백 루프 |
| 포트폴리오 | "혼자 서비스 하나 완성" | "에이전트 팀을 운영해본 경험" |

**멘티가 하네스 엔지니어링을 배우는 방식:**
엔진을 함께 만들면서 멘토가 하네스 세팅(에이전트 환경·정의)을 주도하는 방식을 옆에서 익힌다. 여유가 생기면 직접 기여한다. 처음부터 설계 전체를 맡는 것이 목표가 아니다 — **이 시스템 안에서 에이전트에게 일 잘 시키는 법을 먼저 익히는 것**이 우선이다.

**4주의 실질적 목표:**
1. 에이전트에게 일 잘 시키는 훈련 (목적 + 완수 조건 정의)
2. 혼자 서비스 하나 처음부터 끝까지 만든 경험
3. 포트폴리오 소재 1개

## 1. 하네스 엔지니어란

> **"Humans steer. Agents execute."** — OpenAI, Harness Engineering, Feb 2026

하네스 엔지니어는 코드를 직접 쓰는 사람이 아니라, **에이전트가 올바른 코드를 쓸 수 있는 환경을 설계하는 사람**이다.

### 4가지 핵심 실천

| 실천 | 내용 | MirAI 적용 |
|------|------|----------|
| 레포 = 지식의 SOT | 레포에 없으면 에이전트에겐 없는 것 | 카톡/노션 결정사항 → 레포로 |
| 에이전트 자기검증 | 에이전트가 스스로 결과를 확인할 수 있는 도구 | pytest + CI + 로컬 API 확인 |
| 아키텍처 불변식 기계 강제 | 경계 위반을 사람이 아닌 도구가 잡음 | pre-commit + 린터 |
| 실패 → 환경 개선 루프 | 에이전트 실패 = 환경 설계 실패 신호 | 실패 로그 → AGENTS.md 업데이트 |

### MirAI = 팀의 하네스 실습장

별도 스터디가 없다. MirAI 자체가 교실이다. 엔진을 함께 만들고, 멘토가 세팅하는 하네스 환경을 옆에서 익히면서, 각자 서비스를 혼자 만드는 과정 전체가 실습이다.

## 4. 멘토(이왕원) 역할 — 하네스 엔지니어 + PM

멘토는 **코드를 직접 짜는 사람이 아니라, 팀 전체가 일할 수 있는 시스템을 만드는 사람**이다.

### 하네스 엔지니어로서

| 항목 | 주기 | 구체적 행동 |
|------|------|-----------|
| 환경 설계 | 상시 | AGENTS.md, exec-plans, CI, pre-commit 세팅 및 유지 |
| AC 품질 리뷰 | 매 Issue | "이 AC가 에이전트 실행 가능한가?" |
| 실패 진단 | 발생 시 | "어떤 정보가 레포에 없었나?" 함께 분석 |
| 주간 회고 | 매주 금 | 팀 학습 → AGENTS.md 인코딩 |
| 하네스 시범 | Week 1~2 | 전체 사이클 직접 시연 (환경 설계 → 에이전트 실행 → 검증) — 멘티가 관찰 |

### PM으로서

| 항목 | 주기 | 구체적 행동 |
|------|------|-----------|
| 프로덕트 방향 정의 | 주간 | 이번 주 무엇을 만들 것인가 결정 |
| 스코프 통제 | 상시 | "1번 기능이 안 되면 2번은 없다" |
| 우선순위 판단 | Issue 단위 | 멘티가 올린 이슈 중 이번 주 것과 백로그 분류 |

### 하지 말아야 할 것

| 금지 | 이유 |
|------|------|
| 멘티 대신 코드 짜주기 | 멘티가 직접 에이전트에게 시키는 법을 배워야 함 |
| AC 없는 PR 묵인 | AC 없으면 에이전트도, 리뷰도 없다 |
| 에이전트 실패 시 직접 해결 | "뭐가 레포에 없었나"를 멘티가 찾아야 함 |
| 서비스 범위 확장 묵인 | 자소서→질문 1번 기능이 안 되면 나머지는 의미 없다 |

## 4-1. 멘티 초기 훈련 — 에이전트에게 일 잘 시키기

멘티의 첫 번째 목표는 **에이전트를 도구로 부리는 훈련**이다. 직접 코드를 잘 짜는 것보다, 에이전트가 올바른 코드를 짜도록 지시하는 것을 먼저 익힌다.

### 핵심 훈련: 목적 + 완수 조건 정의

에이전트에게 일을 시킬 때 반드시 두 가지를 명시해야 한다:

| 요소 | 질문 | 나쁜 예 | 좋은 예 |
|------|------|---------|---------|
| **목적 (Why/What)** | 이 작업이 왜 필요한가? 무엇을 만드는가? | "로그인 만들어줘" | "사용자가 이메일+비밀번호로 로그인할 수 있어야 한다. JWT를 반환해야 한다." |
| **완수 조건 (AC)** | 언제 완료됐다고 볼 것인가? | (없음) | "POST /auth/login 호출 시 200 + token 반환. 잘못된 비밀번호는 401." |

### 단계별 진행

| 단계 | 내용 | 시기 |
|------|------|------|
| 1. 관찰 | 멘토가 에이전트에게 일 시키는 방식 관찰 | Week 1 |
| 2. 모방 | 동일한 포맷으로 직접 이슈 작성 + 에이전트 실행 | Week 1~2 |
| 3. 독립 | 혼자 AC 작성 → 에이전트 실행 → 결과 검증 | Week 2~ |
| 4. 기여 | 하네스 환경 개선에 부분 기여 (AGENTS.md, CI 등) | 여유가 생기면 |

## 5. 핵심 위험 요소

| 위험 | 징후 | 대응 |
|------|------|------|
| 기능 구현에 집착 | "일단 코드 짜고 문서는 나중에" | "문서 없는 기능 = 없는 기능" |
